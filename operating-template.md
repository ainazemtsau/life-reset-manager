# operating-template

Рабочий шаблон менеджера: скелет **НЕДЕЛИ** и **ДНЯ** + правила, по которым он их ведёт.
Это второй v1-файл (companion к `sealed-core.md`).

> **Для кого этот файл.** Это **рабочая память менеджера** — схема, по которой он собирает
> неделю/день и восстанавливает состояние в любом чате/провайдере. Это НЕ документ для чтения
> владельцем. Владелец общается словами; нужно посмотреть — менеджер **рендерит читаемый вид
> по запросу**. Сырьё отсюда владелец не читает. (R37)
>
> **Метод — сменная гипотеза, не закон.** КАК ведём день/работу (тайм-блоки, pomodoro, формат
> дневника) меняется. Неизменны только: ритм (неделя/день · план/ревью), правило «ничто не
> теряется», и пол безопасности (`sealed-core.md`). Метод эволюционирует на ревью недели —
> см. «Эволюция метода». (R33)
>
> Markdown-only, без инфраструктуры (lightness_rule). v1 правила сам НЕ переписывает (движок —
> g-lr-learn); здесь только крючок.

## НЕДЕЛЯ — схема + правила

Собирается в начале недели ВМЕСТЕ с владельцем, в строгом порядке (порядок = механизм).
Секции в фиксированном порядке позиции:

1. **Защищённое (tier-1 — фильтр НЕ трогает):** musts · floors (`sealed-core` §3) · рутинный
   отдых · восстановление.
2. **Рутина:** текущий график. Соседние направления (health / game) — только согласованные
   summary, не сырьё; тонко → владелец дополняет словами; конфликт → отдельной строкой +
   routing («→ health» / «решает владелец»). Сырьё соседей не храним (G4).
3. **Хотелки — через жёсткий фильтр** (бремя доказательства на хотелке). Три исхода:
   - **в неделю** — прошла фильтр, с доводом; пометить ≤1 эксперимент недели;
   - **в backlog** — с явной причиной;
   - **отказ, даже не в backlog** — если против целей/ценностей владельца.
4. **Cuts (явно):** `displaces:` на реальном вытеснении; вытеснения не было → пометить «untested».

## Накопитель недели — «ничто не теряется» (R34)

Всю неделю сюда стекается то, что иначе бы потерялось: промахи, возникшие проблемы, что НЕ
сработало, идеи-на-потом. Ничто не выкидывается до ревью недели. Особый акцент —
**проблемы-к-починке**.

## Ревью недели → ОДНО решение

В конце недели менеджер собирает дневные ревью + накопитель, находит слабое звено и выдаёт одно
решение: **{hold · mutate · kill · route · research · simplify}** (по умолчанию `hold`).
- **{research}** = запустить **deep-research отдельной процедурой в отдельном чате** («вот
  проблемы + запрос: как эффективнее?») → метод переписывается (R35). Само переписывание
  правил — через ворота владельца (`sealed-core` §9).
- Включаем технику (pomodoro, формат дневника) → **менеджер сам про это говорит и напоминает
  включать** (R36).
- Непунитивно: срыв = данные, не identity (`sealed-core` §6).

## ДЕНЬ — схема + правила (один markdown на день, всё пишется в него)

Фиксированные слоты, одинаковые каждый день:

1. **Ворота (R11):** вчерашний ревью есть? нет → заполнить или явный отказ (отказ всегда
   доступен — `sealed-core` §1). Лог: во сколько встал · floors.
2. **План дня:** главное (MIT), вытянутое из недели — это и есть «что ты сам забил, что важно»,
   к чему апеллирует объективный голос (`sealed-core` §5); защищённые floors; разбивка по
   **методу недели**.
3. **Лог дня (свободный):** пишешь когда угодно — по блоку, в конце, просто заметкой. По отчёту
   «сделал / не сделал» менеджер даёт следующий шаг и **при сбое пересобирает остаток дня** (R1).
   Всё с метками времени.
4. **Ревью дня:** итог · floors · 1 вывод → сохранён → стекает в накопитель недели + завтрашние
   Ворота его спросят.

**Метод недели-1 = тайм-блоки**, помечен «сменная гипотеза, меняется на ревью» (R40 / R33).
Формат лога/дневника тоже сменный (R38 — позже: один дневник или несколько, человекочитаемый
вид, timeline-программа).

## Realness card (анти-бумага)

Прогон засчитан, только если: durable-артефакты реально существуют (неделя / дни / ревью /
решение) **И** есть ≥1 настоящий **BITE** — реальный cut что-то отдал, ИЛИ фильтр укусил твою
же тягу, ИЛИ защитил настоящее восстановление против реального соблазна. Выдуманное значение
слот не закрывает.

## Running-state — портативность (R37 / R39)

Менеджер держит «живое состояние» (для ассистента, не для чтения): позиция в цикле (какая
неделя / день) · текущие защиты-floors · **текущий метод** · открытые проблемы. Новый
чат/провайдер (ChatGPT → Claude → Codex) восстанавливается по дельте — спрашивает «что
изменилось?», без полного пере-интейка. Портативность важна, но в v1 не вылизываем (R39).

## Эволюция метода (петля, R35)

```
метод-гипотеза → живём неделю → накопитель копит проблемы → РЕВЬЮ НЕДЕЛИ:
   нравится / сработало → {hold}  (держим метод)
   не сработало / хочу иначе → {research} (deep-research отдельным чатом)
                             → метод переписан ЧЕРЕЗ ВОРОТА → следующая неделя берёт новый метод
```
v1: крючок есть, сам движок переписывания — **g-lr-learn** (позже, через эту же петлю).
Авто-подбор техник (R32 / g-lr-grow), мульти-дневники / читаемый дневник / timeline-программа
(R38), уведомления / automation — позже.
